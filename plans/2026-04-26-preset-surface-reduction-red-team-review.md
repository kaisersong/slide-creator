# 2026-04-26 风格支持面收敛方案对抗性评审

评审对象：

- [2026-04-26 风格支持面收敛方案](/Users/song/projects/slide-creator/plans/2026-04-26-preset-surface-reduction-plan.md)

评审目标：

- 不站在“方案制定者”的角度，而站在最苛刻的反对者角度，判断该方案哪里可能把问题看错、做重、或引入新的负担。

---

## 结论先写

这份方案**方向基本正确**，因为它识别出了真正的问题是“支持面和承诺面过宽”，而不是简单的 prompt token 爆炸。

但它也有四个明显风险：

1. 它容易把“稳定性提升”过度归因到 preset 收敛，而忽略真正的质量瓶颈还包括 composition、runtime、title quality、browser QA。
2. 如果只改文案和 picker，不改 routing 与 release gate，方案会变成形式主义。
3. 如果 production 集合过度偏企业风格，可能会弱化 slide-creator 的探索性和吸引力。
4. 在没有 usage data 的情况下，preset tier 名单仍然带较强主观性。

也就是说：

- **方案可推进**
- 但必须按“先收缩默认推荐面，再收缩工程支持面，最后才考虑删资产”的顺序执行
- 绝不能跳过中间层，直接把结论落到“删风格”

---

## 反对意见 1：你可能在解决错误的 bottleneck

最强反对点：

- 当前 `--generate` 并不是把 21 个风格一起塞进上下文。
- render 阶段本来就只读一个 style file。
- 因此把 21 改成 10，未必会显著提升单次生成质量。

这条反对成立的部分：

- 从纯 token 占用看，方案收益确实容易被夸大。
- 如果你把方案卖点写成“删 preset 会显著减少 render prompt”，那是不准确的。

这条反对不成立的部分：

- 方案真正解决的是默认决策面、对外承诺面和工程维护面，不是单次 render token。
- 所以文档和后续实施里，必须坚持这一定义，不能把收益描述错位。

评审结论：

- **允许推进**
- 但所有对外和对内表述都要避免“因为 21 个 preset 会直接撑爆 render context”这种不准确说法

---

## 反对意见 2：你会把产品做得更无聊

最强反对点：

- 21 个 preset 本身就是产品卖点。
- 如果默认面只剩企业感很强的几套，slide-creator 会变得过于保守。
- 用户会觉得“选择更少了、风格更像了、惊喜感没了”。

这条反对成立的部分：

- 如果 Tier 0 + Tier 1 的可见集合过于偏 `Swiss / Enterprise / Data`，确实会产生单一气质。
- 这会伤害品牌感、创意表达和产品差异化。

这条反对给方案的修正要求：

1. 默认活跃面不能只有企业风格。
2. 至少保留：
   - 一个企业稳态风格
   - 一个数据/策略风格
   - 一个 editorial 风格
   - 一个 creative / launch 风格
   - 一个 dev-focused 风格
3. `experimental` 不等于“藏起来不让选”，而是“不默认推荐、但仍可直达”。

补充修正：

- 如果已经有使用量信号，editorial 代表应优先服从使用量，而不是只按抽象风格去重。
- 也就是说，在 `Notebook Tabs` 与 `Paper & Ink` 之间，如果 `Paper & Ink` 使用更高，就不应因为“家族重复”而先把它降级。

评审结论：

- **方案必须保留风格张力**
- 如果最后 visible shortlist 没有 editorial / creative / dev 三类代表，方案应视为不通过

---

## 反对意见 3：你会制造新的双轨复杂度

最强反对点：

- 现在的问题是 preset 太多。
- 你如果再加上 tier、routing、推荐、gate 差异，系统反而会更复杂。

这条反对成立的部分：

- 如果 tier 只是文档标签，而没有真正替代旧复杂度，那就是“复杂度叠加”，不是“复杂度转移”。
- 比如：
  - picker 还是 21 个平铺
  - validate 还是所有 preset 等权
  - release 还是全量盯
  - 只是 README 改成了分层
  那基本没有收益。

方案必须满足的最低条件：

1. tier 进入 metadata，而不是停留在 PPT 概念层。
2. 默认推荐逻辑必须真的引用 tier。
3. release gate 必须真的优先保证 Tier 0。
4. 非 Tier 0 的失败，不再默认等价于 production regression。

评审结论：

- **如果只改文案，不改系统行为，则不建议做**
- **只有当 tier 进入 routing / validate / release 流程，方案才成立**

---

## 反对意见 4：没有 usage data，tier 名单会很主观

最强反对点：

- 你现在给的分类，本质上还是工程判断，不是用户使用数据。
- 这可能会把其实很受欢迎的风格误降级。

这条反对成立。

因此，方案应该被修正为：

- **第一阶段只定义 tier 和默认推荐面**
- **不做不可逆删除**
- **不把 Tier 2 直接从产品里消失**

同时要加一个决策约束：

- 真正进入 archive candidate 的 preset，至少要同时满足：
  - 与主力风格高度重叠
  - 维护成本高
  - 在多个版本周期内没有被提升优先级

评审结论：

- 当前阶段可以做 tier
- 当前阶段不应做激进删减

---

## 反对意见 5：你可能会掩盖真正的质量问题

最强反对点：

- 即使把默认 visible preset 从 21 压到 8，以下问题依然会存在：
  - runtime bug
  - reveal / notes / wheel 之类共享壳子 bug
  - 标题编排问题
  - composition 走样
  - narrative 与 renderer 错配

这条反对成立。

因此必须明确：

- 方案不是质量问题的替代修复
- 它只是让团队更容易把质量精力集中在主力风格上

如果有人把这份方案理解成“只要少几个 preset，质量就自然变好”，那就是误用。

评审结论：

- **这是工程聚焦方案，不是质量魔法**

---

## 修正后的可接受版本

经过 red-team 后，方案应该收敛成下面这个更硬的版本：

### 1. 不删资产，先收缩默认活跃面

- 21 个 preset 保留
- 第一阶段默认 visible shortlist 可先只保留核心 4 个
- 所有 preset 仍可被显式点名

### 2. 生产级集合只定义，不夸大

- `Swiss Modern`
- `Enterprise Dark`
- `Data Story`
- `Blue Sky` 作为 starter / product-demo 特例

不要把 Tier 1 也写成“稳定等价”。

### 3. 把 Tier 真正写进系统

至少要落到：

- preset metadata
- picker 分组
- default recommendation
- release gate

### 4. 先跑两个版本周期，再决定是否扩面或归档

只有在 tier 模型跑过一段时间后，才判断是否做物理删减。

---

## Go / No-Go 标准

### Go

满足以下条件时，可以推进：

1. 实施目标是“收缩默认支持面”，不是“立即砍资产”
2. Tier 进入真实系统行为，而不只是文案
3. 第一阶段允许只守核心 4 个；若进入扩面阶段，默认可见集合必须保留企业、数据、editorial、creative、dev 五类代表
4. 物理归档延后到后续版本周期

### No-Go

出现以下任一情况时，不建议推进：

1. 只改 README，不改 picker / routing / gate
2. 把所有非主力 preset 都从用户视野中拿掉
3. 把“上下文优化”错误表述为主要收益
4. 试图在没有额外证据的情况下立刻删除一大批 preset

---

## 最终评审结论

这份方案**可以推进，但只能按保守版本推进**：

- 先做支持分层
- 再做默认推荐收缩
- 再做系统层承认
- 最后才讨论删除

如果按这个顺序执行，它会提升稳定性和输出质量。  
如果跳过中间层，直接用“删 preset”解决问题，副作用会大于收益。
